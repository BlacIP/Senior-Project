LocalLink Marketplace

Video Walkthrough Script — CSE 499 Senior Project
Next.js · Neon DB · Drizzle ORM · Paystack · Cloudflare R2

Section 1 — Project Requirements 2–3 min

Begin by briefly introducing the project: LocalLink is a cross-platform web marketplace that connects local vendors with nearby customers. The application was built with Next.js and deployed on Vercel, using Neon DB (serverless PostgreSQL) for persistent storage and Paystack for payment processing.

# Requirement Description Type
1 User Account Management Customers and vendors sign up, log in securely, and access role-specific dashboards. Core
2 Vendor Product Listings Vendors create, edit, delete, and manage product listings with images, price, stock, and category. Core
3 Product Search & Filtering Customers browse products and filter by keyword, category, vendor city, and price range. Core
4 Cart & Checkout Customers add products to a cart, review their order, and complete payment via Paystack. Core
5 Order Management Customers view order history and track status; vendors update orders from pending to delivered. Core
6 Location-Based Discovery Geolocation API auto-detects the customer's delivery coordinates at checkout. Enhancement

Section 2 — Product Walkthrough 8–10 min

Walk through the live running application in the order below. Each step maps to a numbered requirement.

Requirement 1 — User Account Management

  1. Navigate to /register. Fill in name, email, password, and select the Customer role. Submit.
  2. After redirect, navigate to /login and sign in with the new account.
  3. Point out the customer navigation bar (Marketplace, Cart, Orders, Profile).
  4. Log out, then register again — this time select the Vendor role.
  5. Log in as the vendor and show the vendor dashboard at /vendor with its different layout (Products, Orders, Store tabs).
  6. Explain that role assignment is enforced on both the frontend and backend API routes.

Requirement 2 — Vendor Product Listings

  1. From the vendor dashboard, click Add New Product to go to /vendor/products/new.
  2. Fill in: product name, description, price, stock quantity, category, and upload a product image.
  3. Submit and confirm the product appears in the vendor's product list.
  4. Click Edit on the product — change the price and save.
  5. Demonstrate the Available / Out of Stock toggle.
  6. Click Delete on the product and confirm it is removed from the list.

Requirement 3 — Product Search & Filtering

  1. Log in as the customer and navigate to /marketplace.
  2. Type a keyword in the search bar — point out the product list updating in real time.
  3. Open the Category dropdown and select a category to narrow results.
  4. Use the City filter to show vendor-location-based filtering.
  5. Adjust the Price Range slider and show results narrowing.
  6. Click on any product card to open the product detail page at /marketplace/products/[slug].
  7. Show the vendor name, description, price, stock level, and Add to Cart button on the detail page.

Requirement 4 — Cart & Checkout

  1. On the product detail page, click Add to Cart. Show the cart icon counter update in the navbar.
  2. Add one or two more products from different vendors.
  3. Navigate to /cart.
  4. Increase the quantity of one item and remove another item to demonstrate cart management.
  5. Fill in a delivery address manually in the text field.
  6. Alternatively, click Detect My Location — show the coordinates auto-filling the field (Requirement 6).
  7. Click Checkout — a Paystack payment dialog appears. Complete the demo payment.
  8. Confirm the success message and redirect to the orders page.

Requirement 5 — Order Management

  1. On /orders as the customer, show the newly created order with its Pending status and the delivery progress bar.
  2. Show order items broken down by vendor, with price totals.
  3. Switch to the vendor account and go to the Orders tab on the vendor dashboard.
  4. Find the incoming order. Use the status dropdown to change it to Confirmed, then Shipped, then Delivered.
  5. Switch back to the customer account and refresh /orders — show the status has updated and the progress bar has advanced.
  6. Click Confirm Delivery to mark the order as complete from the customer side.

Section 3 — Code Walkthrough 6–8 min

Each sub-section below has a SHOW cue (what to put on screen) and a SAY cue (the spoken narration). You do not need to read every line of code — scroll slowly and let the narration guide the viewer's eye.

Architecture Overview

Show → project folder structure in the IDE (collapse node_modules)

"Before we dive into individual files, let me quickly walk through how the project is structured. We're using a Next.js monorepo, so the frontend and backend live in the same codebase. Inside src/app you'll find all the pages and the API routes. src/components holds the interactive client-side UI — things like the marketplace grid and the cart page. src/server is where all the pure server-side business logic lives — authentication, orders, vendor operations. src/db has the database schema and client, and src/lib holds shared utilities like cart helpers, validators, and the image upload client. This separation keeps things clean: the UI never directly touches the database — it always goes through the service layer."

1. Database Schema — src/db/schema.ts

Show → open schema.ts, scroll slowly from top

"This file defines the entire database using Drizzle ORM. At the very top you can see two enums — userRoleEnum with values customer, vendor, and admin, and orderStatusEnum with the five stages an order moves through: pending, confirmed, shipped, delivered, and completed. These enums are enforced at the database level, so the app can never write an invalid value."

Show → scroll to userProfiles table, highlight the role column

"The userProfiles table stores every user. Notice the role column — it defaults to 'customer', but when someone registers as a vendor we set it to 'vendor'. That single column is what controls which dashboard a user sees and which API routes they can call."

Show → scroll to products table, then orders and orderItems

"The products table has a foreign key pointing back to vendorProfiles, so every product is always owned by a specific vendor. Down here in orders, you can see we also store the delivery address and GPS coordinates alongside the payment reference from Paystack. And orderItems is the join table that links a single order to one or more products, storing the quantity and the unit price at the time of purchase — so even if the vendor later changes the price, the historical order stays accurate."

2. Authentication — src/lib/auth.ts & src/server/auth-service.ts

Show → open auth.ts

"Authentication is powered by Neon Auth, which is the auth layer built into the same Neon database we use for our data. This file initializes it. The first line — import 'server-only' — is a Next.js guard that makes the build fail if this module ever gets bundled into the client-side JavaScript. That's important because this file holds our cookie secret, and it should never ship to the browser."

Show → open auth-service.ts, scroll to registerUser (line 56)

"Now let's look at how registration works. registerUser first validates the incoming data against a Zod schema, then calls auth.signUp.email to create the user in Neon Auth. Once that succeeds and we get back an auth user ID, we call upsertUserProfile to insert a matching row in our own userProfiles table — and this is where we attach the role the user selected on the registration form. So we have two records in sync: one in Neon Auth for credential management, and one in our own database that carries the role and any app-specific data."

Show → scroll to loginUser (line 84)

"Login works similarly. We call auth.signIn.email which validates the password and sets the session cookie automatically. Then we look up the corresponding profile in our own database to confirm the account exists and to return the user's role to the frontend — that's what decides which dashboard the user lands on after login."

3. Vendor & Product Service — src/server/vendor-service.ts

Show → open vendor-service.ts, scroll to requireVendorSession (line 18)

"Every vendor operation starts with this guard function — requireVendorSession. It reads the current session cookie, looks up the user profile, and checks that the role is 'vendor'. If either check fails it returns null, and the calling function immediately exits. This means it is impossible to call any vendor write operation — create, update, or delete — without being authenticated as a vendor. The check happens on the server, so it can't be bypassed from the browser."

Show → scroll to createVendorProduct (line 148), highlight the vendorId insert

"When a vendor creates a product, we first fetch their vendor profile using the session, then insert the product with vendorId set to that profile's ID. Notice on the update and delete functions — the WHERE clause always includes both the product ID and the vendor ID. That means a vendor can only ever modify their own products, even if they somehow knew another product's ID."

Show → scroll to listProducts (line 250)

"This is the query that powers the public marketplace. It does a left join between products and vendorProfiles so each product comes back with its vendor's business name and city attached. We filter to only return available products, order by newest first, and cap it at 48 results. The filtering by keyword, category, and city that you saw in the UI happens on the client side against this result set."

4. Cart — src/lib/cart.ts

Show → open cart.ts, highlight getCartStorageKey and writeStorage (lines 20–33)

"The shopping cart is stored entirely in the browser's localStorage — there's no server round-trip every time you add or remove an item. To support multiple users on the same device, the storage key includes the user's ID. So if you're logged in as one customer and someone else logs in on the same browser, they each get their own separate cart."

Show → scroll to addCartItem (line 57), highlight the stockQuantity cap

"When you add an item, we check whether it's already in the cart. If it is, we increment the quantity — but we cap it at stockQuantity, so you can never add more than what's actually available. And at the bottom of the writeStorage function you can see we dispatch a custom event called locallink-cart-changed. The navbar listens for this event and immediately updates the cart count badge — no page reload needed."

5. Order Flow — src/server/order-service.ts

Show → open order-service.ts, scroll to createOrder (line 86)

"When a customer completes checkout, this function runs on the server. It first looks up the customer profile, then iterates over every item in the cart to fetch the current price from the database — we never trust the price sent from the browser, because a malicious user could tamper with it. We recalculate the total on the server from the database price. Then we insert a single orders row and one orderItems row per product. Notice this line — if a payment reference came in from Paystack, the order status starts as 'confirmed' and payment status as 'paid'. Without a payment reference it stays 'pending'."

Show → scroll to updateVendorOrderStatus (line 223)

"When a vendor changes an order's status, we don't just check that they're a vendor — we also verify that their products are actually in that specific order. We do a join through orderItems to products filtered by the vendor's ID. If they don't own any item in that order, the function returns 'not found' and the status doesn't change. That prevents one vendor from accidentally — or maliciously — updating another vendor's orders."

Show → scroll to listCustomerOrders (line 155), highlight the groupOrderRows call

"Fetching orders for the customer page involves a multi-table join — orders, order items, products, and vendor profiles all in one query. Because a single order can have multiple items, the raw SQL returns one row per item. The groupOrderRows helper re-groups those flat rows by order ID, building a nested structure where each order has an array of its items. That's what the UI consumes to render the order breakdown you saw in the product walkthrough."

6. Image Uploads — src/lib/r2.ts

Show → open r2.ts, highlight getR2Client and uploadProductImage

"Product images are stored in Cloudflare R2, which is an S3-compatible object storage service. This file sets up the S3 client pointed at Cloudflare's endpoint using our account ID and access keys from environment variables. When a vendor uploads a photo, the uploadProductImage function validates the file — checking type and size — then generates a unique storage key using a UUID under a path like vendor-products / vendorId / filename. It uploads the file directly from the server to R2 using a PutObjectCommand, then returns both the key and the public URL. That URL is what gets saved to the imageUrl column in the database and served to customers browsing the marketplace."

Recording Tips

  • Start the dev server before recording: npm run dev
  • Seed demo data if needed: node scripts/seed-demo-data.mjs
  • Use two browser profiles simultaneously — one logged in as a customer, one as a vendor — so you can switch roles without logging out mid-recording.
  • Record at 1920×1080; set browser zoom to ~110% so text is legible in the video.
  • Mute browser notifications before recording.
